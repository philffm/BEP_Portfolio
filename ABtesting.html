Week 5: Explorative Prototyping: AB-Testing

5.1. Introduction: The assignment

This week’s prototype was meant as an excursion to AB-testing. The overall subject dealt with visual hierarchy, thus we were able to choose between different topics: Gaze following, Scanning patterns or color – however, we were free to make any other suggestion. As I stumbled over a related issue in my last week’s ID2-project, where we defined the information architecture and design for our own online shop, I asked my lecturer whether I could conduct an AB-Test for one particular page of this online shop.

5.2. Preparing the test

In the end, we came to the agreement, that the following hypothesis could deliver interesting insights: “Adding a “bottom buy box” to a product page can lead both to higher conversions & shorter time to action.”

The online shop was already created in Adobe XD. In order to prepare the prototype for my AB-Test, I followed the following steps:

  1. Create second version of desired appearance (incl. a bottom “add-to cart” button, that gets visible, once a certain scroll state is reached (what in particular didn’t work out, but I had the design for now)
  2. Took the base design and exported it using the “Velara 3 Web Export”-plugin in Adobe XD. - This resulted in a messy html & css files and a bunch of even messier assets.
  3. Reorganized layers in XD, to get a cleaner HTML structure – Learning: Reorganizing layers in XD worked quite well and resulted in a reasonably bettter structure in the exported HTML & CSS.
  4. Defined divs where needed, told them to float on the top layer and stick to the top/bottom
  5. Included JavaScript file, that randomizes the played out prototype version, tracks the scroll state and shows the altered bottom buy button if top buy button is out of vision
  6. Included Google Tag Manager / Analytics script, defined all element triggers, variables, event trackers in Google Tag Manager
  7. Tested every possible action whether tracking in Google Analytics works

5.4. Observation: Participants and insights

As mentioned on the setup document, my conducted testing consists of two sources of data:

  1. Quantitative user testing
    Live URL with instructions - possibly big test group
  2. Qualitative User testing
    Test-setup in class

When I conducted the qualitative user testing in class, I began using the URL where a randomized test version is played out. However, the randomize function returned the version A, which I was less interested to test, way more often. That’s why I decided to include a custom variable, so the accessed URL decides which test is shown. That gave me the ability to better control the trial in class. However, tests after the trial (Google Analytics) showed that the randomize-function factually worked and showed variant A just as often as variant B.

The user trial in general showed me yet again, how much value qualitative user trialing can actually add. Plenty of my participants decided to take the top buy button instead of the bottom buy button as predicted in my hypothesis. Of course, I was curious about that: So I asked them not, why they didn’t go for the bottom buy box, but asked whether they actually saw a different way of putting the article to the shopping cart.

I got the following answers:

  • Oh, it appears during scrolling – I was looking at the content
  • I perceived it – but I saw the top buy button before, so I knew I just had to scroll up again to add the article
  • Maybe make it bigger? Or floating?

5.5. Gathering data in Google Analytics

User trialing in class was not enough for me – so I shared the prototype with some friends in order to gain further insights. My main insight was: They behaved similar – but without the user trialing, I would probably still be looking for the needle in the haystack, trying to optimize a fail-proof concept. However, with the gained user insights, a new direction was paved: Version C with a possibly bigger, floating buy box.

5.6. Conclusion

This assignment showed me yet another time, how important user trialing can be – not only in order to validate hypothesizes, but also to gain new insights, that can be used to improve the prototypes being tested and pitch it towards the best possible solution.
